# 邸报 Dibao MVP PRD

## 文档目的

本文定义邸报 Dibao 第一版 MVP 的产品范围、核心用户流程、页面清单、功能优先级与验收标准。

MVP 的目标不是做完整 RSS 生态替代品，也不是追求复杂推荐系统，而是验证一件事：

> 用户能否把自己的 RSS 信源交给邸报，并明显感觉到“在我的关注源里，排序比时间线更懂我”。

## MVP 定位

邸报 MVP 是一个单用户、自托管、服务端优先的个人 RSS 推荐阅读器。

它需要做到：

- 用户可以导入或添加 RSS / Atom 订阅源。
- 系统可以定时抓取文章、抽取正文并保存。
- 用户可以在 Web / Mobile Web / PWA 中阅读文章。
- 系统可以基于 embedding、用户行为和规则排序未读文章。
- 用户可以看到轻量推荐解释，并通过行为反馈影响后续排序。
- 数据默认保存在用户自托管环境中。

## 非目标

MVP 明确不做：

- 多用户账户体系
- 团队协作
- 官方托管服务
- 社交关系、关注、评论、转发
- 平台外内容推荐
- 原生 iOS / Android App
- 纯移动端本地计算模式
- 付费墙解析和反爬对抗
- 复杂机器学习训练平台
- 通用文章归档系统
- 兼容所有 RSS 阅读器私有导出格式

## 目标用户

MVP 首先服务愿意部署自托管服务的个人用户。

典型用户：

- 有大量 RSS 订阅源。
- 每天新增未读文章约 300～400 篇。
- 希望保留对信源的控制权。
- 不满足于传统 RSS 时间线。
- 可以接受配置 Docker、OPML、embedding provider 等基础设置。

长期目标是让普通用户也能使用，但 MVP 不追求“零技术门槛”。

## 成功标准

MVP 成功的最低标准：

- 用户可以在 30 分钟内完成 Docker 部署和首次配置。
- 用户可以导入 OPML 并成功抓取文章。
- 用户可以在桌面和手机浏览器中完成主要阅读流程。
- 用户可以配置至少一种外部 embedding provider。
- 未配置外部 provider 时，系统仍可通过内置 fallback 或时间线模式可用。
- 首页推荐流能基于用户行为发生可感知变化。
- 用户可以查看每篇推荐文章的主要排序原因。
- 用户可以导出 OPML，不被锁死在邸报中。

## 核心用户流程

### 首次启动

1. 用户部署邸报服务。
2. 用户打开 Web App。
3. 系统进入首次设置向导。
4. 用户设置单用户访问密码。
5. 用户导入 OPML 或手动添加第一个 RSS 源。
6. 用户选择 embedding provider：
   - 内置基础模型
   - Ollama / 外部本地 provider
   - OpenAI-compatible provider
   - 暂不配置，仅使用基础排序
7. 系统测试 provider 可用性。
8. 系统开始首次抓取。
9. 用户进入文章列表。

### 日常阅读

1. 用户打开邸报。
2. 首页显示个性化排序后的未读文章。
3. 用户可以切换：
   - 推荐
   - 最新
   - 收藏
   - 稍后读
   - 来源
   - 搜索
4. 用户点击文章进入阅读页。
5. 阅读行为被记录：
   - 打开
   - 停留
   - 滚动进度
   - 标记已读
   - 收藏
   - 稍后读
   - 不感兴趣
6. 后台根据行为更新画像和排序。

### 理解推荐

1. 用户在文章列表中点击解释入口。
2. 桌面端展示 Popover，移动端展示 Bottom Sheet。
3. 系统展示 3～5 条主要原因：
   - 匹配兴趣
   - 来源权重
   - 新鲜度
   - 重复惩罚
   - 负向兴趣扣分
4. 用户可以据此选择阅读、收藏或不感兴趣。

### 管理订阅源

1. 用户进入订阅源管理。
2. 用户可以：
   - 添加 RSS / Atom URL
   - 导入 OPML
   - 导出 OPML
   - 修改源名称
   - 设置分组
   - 启用 / 禁用订阅源
   - 手动刷新
   - 查看最近抓取状态
3. 抓取失败时显示错误原因和重试入口。

### 配置推荐能力

1. 用户进入推荐设置。
2. 用户查看当前 embedding provider 档位。
3. 用户可以配置：
   - 内置基础模型
   - Ollama
   - OpenAI-compatible API
   - 自定义 HTTP provider
4. 系统显示：
   - 模型名称
   - 向量维度
   - 连接状态
   - 最近错误
5. 用户更换模型时，系统提示需要重建 embedding index。

## 页面清单

### 首次设置向导

用途：帮助用户完成最小可用配置。

包含：

- 欢迎页
- 单用户访问密码设置
- OPML 导入 / 添加订阅源
- embedding provider 选择
- provider 连通性测试
- 首次抓取进度

### 推荐首页

用途：日常主要入口。

包含：

- 推荐文章列表
- 排序方式切换
- 未读计数
- 文章状态标记
- 收藏 / 稍后读 / 不感兴趣
- 推荐解释入口
- 手动刷新入口

### 最新文章

用途：保留传统 RSS 时间线。

包含：

- 按发布时间排序
- 来源筛选
- 已读 / 未读筛选
- 分组筛选

### 文章阅读页

用途：沉浸阅读。

包含：

- 标题
- 来源
- 发布时间
- 原文链接
- 正文
- 收藏
- 稍后读
- 标记已读 / 未读
- 不感兴趣
- 阅读样式设置入口

阅读设置包括：

- 字号
- 行高
- 段间距
- 阅读宽度
- 背景色
- 字体选择

### 订阅源管理

用途：管理 RSS 输入。

包含：

- 源列表
- 分组
- 添加源
- 导入 OPML
- 导出 OPML
- 启用 / 禁用
- 手动刷新
- 抓取状态
- 错误提示

### 搜索

用途：在本地文章库中检索。

MVP 搜索基于 SQLite FTS5。

包含：

- 关键词搜索
- 来源筛选
- 时间筛选
- 收藏 / 稍后读筛选

### 收藏与稍后读

用途：保存用户主动标记的文章。

包含：

- 收藏列表
- 稍后读列表
- 批量标记已读
- 移除收藏 / 稍后读

### 设置

用途：管理系统级配置。

包含：

- 访问密码
- embedding provider
- 推荐设置
- 阅读设置
- 数据保留策略
- OPML 导入 / 导出
- 任务状态
- 系统信息

## 推荐与反馈需求

MVP 的推荐系统必须满足：

- 推荐只在用户订阅源内部发生。
- 新文章可以先进入列表，不因 embedding 未完成而不可见。
- 首页优先展示预计算排序结果。
- 未 embedding 的文章使用基础排序兜底。
- 用户行为必须能影响后续排序。
- 推荐解释必须来自排序时保存的 explain payload。

行为信号初始定义：

```text
打开文章：轻度正信号
阅读较长时间：中度正信号
读完：较强正信号
收藏：强正信号
稍后读：中度正信号
不感兴趣：强负信号
只曝光未点击：弱信号，不直接视为强负反馈
```

## 数据保留需求

MVP 默认文章保留 60 天。

用户可以在设置中调整：

- 保留天数
- 是否保留收藏文章
- 是否保留稍后读文章

删除旧文章时：

- 删除文章正文。
- 删除文章 embedding。
- 删除全文索引。
- 保留必要行为摘要。
- 保留已经沉淀到用户画像中的长期影响。

## 隐私与数据所有权

MVP 必须满足：

- 数据默认保存在用户自托管环境。
- 不提供官方托管。
- 不上传使用数据。
- 外部 embedding provider 由用户主动配置。
- 当使用云端 provider 时，界面必须提示文本会发送给第三方服务。
- 数据库结构和内部数据格式需要文档化。

## 移动端要求

MVP 不做原生 App，但 Web App 必须从一开始适配移动端。

要求：

- 手机浏览器可用。
- 支持添加到主屏幕。
- 文章列表和阅读页在窄屏下可用。
- 推荐解释在移动端使用 Bottom Sheet。
- 主要操作适合触摸。
- 不依赖 hover 才能完成核心流程。

## 验收清单

MVP 可以认为完成，当以下场景全部通过：

- 用户部署 Docker 服务后可以打开 Web App。
- 用户完成首次设置并设置访问密码。
- 用户导入一个 OPML 文件。
- 系统成功抓取多个订阅源。
- 用户可以阅读文章并标记已读。
- 用户可以收藏、稍后读和标记不感兴趣。
- 用户可以搜索文章。
- 用户可以配置一个 OpenAI-compatible provider 并通过测试。
- 系统可以生成 article embedding。
- 系统可以写入 sqlite-vec 索引。
- 系统可以根据行为生成正向 / 负向兴趣簇。
- 首页推荐排序会随用户行为变化。
- 用户可以查看推荐解释。
- 用户可以导出 OPML。
- 手机浏览器可以完成阅读和反馈流程。
